# Portfolio & Resume Project Roadmap

The Maverick-canonical milestone view for the public portfolio site and the resume / cover-letter materials. Two epics, two distinct cadences. Read [`PROJECT_CONTEXT.md`](PROJECT_CONTEXT.md) for system shape and the Astra/Netwise naming rule.

## Audience

Primary: **technical hiring managers and engineering peers** evaluating Brooks for software engineering roles. Recruiters land here too, but they are downstream readers — the bar is "what does an engineer who's seen 1000 portfolios think when they land here in 30 seconds."

The portfolio is career proof: who Brooks is, what he builds, what proof exists, which projects deserve attention. The resume is the formal artifact accompanying actual job applications.

Public site: **`brookswimer.com`** (per CNAME). The legacy `brookswimer.dev` reference in `og:url` is stale metadata and should be treated as such.

## Active milestones

### M1 — Portfolio reaches "compelling on first glance"

**Why.** This is the substantive content of the existing "Making Portfolio Pop" planning baseline (preserved in [`epics/portfolio.md`](epics/portfolio.md) and [`plans/making-portfolio-pop.md`](plans/making-portfolio-pop.md)). The portfolio site is functional but reads like a label rather than a positioning statement; project cards need stronger hooks; titles need consistent casing; flagship projects (Maverick, Astra, SyncSonic) deserve flagship treatment.

**Scope.** All seven slices from the recovered planning baseline:
1. Hero positioning + anchor nav
2. Project card hierarchy (one-line hooks, fix placeholder titles)
3. Restrained visual system (white-on-dark + restrained blue accent; no new frameworks)
4. About / Experience leading with differentiators
5. Detail pages for the flagship projects (Maverick, SyncSonic, Astra, Kitbash, FoodPal)
6. Mobile + accessibility (fixed-height behavior, aria-label, focus-visible)
7. Verification: local serve, desktop + mobile inspection, link check

**Success signal.** A technical hiring manager who has never met Brooks lands on `brookswimer.com`, takes 30 seconds, and comes away with a clear answer to "what does this person build, and is the proof real?"

**Lane.** [`portfolio`](epics/portfolio.md). Branches from `portfolio` epic branch.

### M2 — Featured project ordering reflects current proof

**Why.** Today's project inventory mixes flagship and classic without deliberate ordering. Per operator decision: featured order is **Maverick → Astra → SyncSonic** (these are the three current flagship proofs of capability). Older projects stay visible (no neglected archive) but should not dominate above-the-fold.

**Scope.** `index.html` ordering, project card prominence, hero featuring. Could land as part of M1's slice 2 (project card hierarchy) — listed as its own milestone because the operator may decide on featured ordering separately from the broader visual rework.

**Success signal.** Featured projects render in Maverick → Astra → SyncSonic order with proportional visual weight. Older projects (Kitbash, FoodPal, 3D models, etc.) read as part of a curated body of work rather than a chronological dump.

**Lane.** [`portfolio`](epics/portfolio.md).

### M3 — Resume master inventory extraction

**Why.** Resume work needs a durable inventory to tailor from — not "open the PDF and edit." Master inventory is the prerequisite for any tailored variant or cover letter. Resume epic is **lower priority** than portfolio for now (operator's stated framing) — when active, the inventory must come first.

**Scope.** Extract structured career facts from `Brooks Wimer Resume.pdf` into `resume/MASTER_INVENTORY.md`. Format: roles, dates, accomplishments, skills, projects, durable claims with evidence pointers. Single source of truth for any future tailored output.

**Success signal.** A tailored resume for a specific role can be generated by selecting from the master inventory — no need to re-derive facts from the PDF or invent numbers.

**Lane.** [`resume`](epics/resume.md). Branches from `resume` epic branch.

### M4 — First tailored resume variant + cover letter generated from inventory

**Why.** Proves the master inventory pipeline by producing a real tailored output for a real role.

**Scope.** Operator picks a target role / role family. Generate a tailored resume variant + cover letter by selecting from the master inventory and applying role-specific framing. Save outputs as inspectable artifacts in the repo. No invented claims.

**Success signal.** The operator submits the tailored materials to a real application and the materials feel like materials, not like LLM output — concise, specific, grounded in actual experience.

**Lane.** [`resume`](epics/resume.md).

## Priority order

1. **M1 portfolio polish** — substantive content work; biggest leverage for the audience.
2. **M2 featured ordering** — small but high-visibility; can land alongside M1.
3. **M3 master inventory extraction** — required floor for resume work; do this before M4.
4. **M4 tailored variant** — only meaningful after M3 lands.

Resume epic is **lower priority right now** than portfolio. M3 + M4 are deferred until the operator decides to invest in formal application materials. The tooling pipeline (master inventory → tailored output) should already exist when the time comes; M3 is exactly that prep.

## Out of scope

- Inventing numbers, fabricating claims, or stretching scope on any project.
- Removing older projects just because they're old. Curate ordering, don't prune the archive.
- Adding a new frontend framework or new deploy dependency for the portfolio. Direct edits to `index.html` and `assets/css/main.css` (with SASS coordinated if used).
- Performance-review-style writeups about coworkers in the resume / cover letter materials.
- Touching `CNAME`, GitHub Pages settings, or the resume PDF link in any of these milestones.
